# Bifrost 项目目标与需求分析

## 核心问题

**问题**: AI 网关根据请求路由不同模型的需求是否真实？为什么缺少开源实现？

## 需求真实性评估

### ✅ 真实需求场景

1. **个人/小团队成本优化**
   - 根据任务类型选择性价比最高的模型
   - 简单任务用 DeepSeek ($0.14/M tokens)
   - 复杂推理用 Kimi K2 ($1/M tokens)
   - 典型节省: 70-85% 成本

2. **企业级统一接入**
   - 统一 API 接口
   - 审计和成本控制
   - 多模型备份和容灾

3. **应用开发者**
   - SaaS 产品需要动态平衡成本和质量
   - 不同用户套餐使用不同模型层级

### ❌ 不会用的人群

- 直接用 ChatGPT 的普通用户
- 财大气粗只用 Claude Opus 的企业
- 不在意成本的研究者
- 需要极简体验的用户

## 为什么开源方案少？

### 1. 已有商业解决方案占领市场

| 产品 | 定位 | 价格 | 局限性 |
|------|------|------|--------|
| **OpenRouter** | 聚合网关，100+ 模型 | 免费 + 按量 | 不支持自定义路由规则 |
| **LiteLLM** | 统一接口转换 | 开源 | 路由功能弱，主要做格式转换 |
| **Portkey.ai** | 企业级可观测性 | $49/月起 | 商业闭源，价格贵 |
| **Helicone** | 可观测性平台 | $20/月起 | 专注监控，路由功能少 |
| **Cloudflare AI Gateway** | 基础网关 | 免费 | 功能简单，无智能路由 |

### 2. 需求相对垂直

- 大多数开发者直接用单一提供商（OpenAI/Anthropic）
- 需要智能路由的是高级用户，可以自己写脚本
- 企业客户倾向购买商业服务（SLA 保障）

### 3. 技术实现有隐藏复杂度

```
- 流式响应的正确处理和故障切换
- 不同提供商的 API 差异（格式、参数、限流）
- 成本计算的实时性和准确性
- 状态管理和监控
- 插件系统的设计和维护
```

## Bifrost 的差异化优势

### 1. 已实现的技术亮点

**CEL 表达式路由**
```json
"cel_expression": "headers[\"x-modality\"] == \"vision\" && headers[\"x-tier\"] == \"quality\""
```
- 比简单的 if-else 灵活
- 声明式配置，易于维护
- 支持复杂条件组合

**Go 插件系统**
```
plugins/classifier/classifier.so
```
- 可动态扩展分类逻辑
- 性能优秀
- 不需要重启服务

**持久化可观测性**
```
config.db - 25MB (配置历史)
logs.db   - 720KB (请求日志)
```
- 完整的请求追踪
- 成本分析数据
- 历史配置版本

### 2. 规划中的杀手功能

**P3: 三层内存架构**
- 本地嵌入模型 (Ollama)
- 按需检索，减少 context 注入
- 标准化 `memory/` + `vault/` 结构

**P4: 编排器/工作器分离**
- 主模型规划，子代理执行
- 自动降级到 economy 层
- 成本优化可达 10x

**P6: 语义缓存**
- `x-bf-cache-key` header
- 相同提示复用响应
- 大幅降低重复请求成本

**P7: 对话压缩**
- 自动总结历史会话
- 降低上下文窗口成本
- 保持长期记忆

## 目标用户画像

### 主要用户

**AI Power Users**
- 特征: 有技术背景，成本敏感，追求效率
- 需求: 多模型切换，成本可控，配置灵活
- 例子: 你自己使用 OpenClaw 的场景

**独立开发者/小团队**
- 特征: 预算有限，需求多样
- 需求: 简单配置，立即省钱，自托管
- 痛点: OpenAI 太贵，DeepSeek 不够强

**企业 AI 平台团队**
- 特征: 需要统一管理多个 AI 应用
- 需求: 审计、成本分摊、合规性
- 痛点: 多个团队各自接入，成本失控

### 次要用户

**SaaS 产品开发者**
- 不同用户套餐使用不同模型
- 免费用户用 economy，付费用户用 quality

**AI 实验者**
- 快速测试多个模型效果
- 无需改代码切换 provider

## 结论

### 这不是伪需求，但是小众需求

**市场定位**: Niche but valuable (小众但有价值)

**存在理由**:
1. 现有开源方案不能满足高级路由需求
2. 商业方案太贵，不适合个人和小团队
3. 技术栈扎实 (Go + CEL + 插件系统)
4. 有真实用例 (OpenClaw 集成)

**优先级**:
- ✅ P1: 完成 OpenClaw 路由契约
- ✅ P2-P3: 内存优化，降低 context 成本
- 🚧 P4-P6: 编排分离、缓存，这些是差异化功能
